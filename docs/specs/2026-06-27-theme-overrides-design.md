# adorn — theme.toml overrides scheme.toml (design)

> **Status:** design, approved 2026-06-27. Engine prerequisite for the adorn
> shell (see the shell spec). Implemented via a follow-on plan.

## Summary

Today a theme's `theme.toml` carries one meaningful key — `scheme = "<name>"` —
selecting which scheme's color derivation to use. This change makes `theme.toml`
a full **override layer** over the chosen scheme's `scheme.toml`: any derivation
section a theme declares (`[mood]`, `[hues]`, `[fixed]`, `[[list]]`) overrides the
scheme's corresponding values, per key. The scheme is the reusable *base*; the
theme is a *per-instance override* on top of it.

A scheme stays a reusable color *system*; a theme can now nudge that system for
its one wallpaper without forking a whole new scheme.

## Motivation

- A user wants "the `oceanic` scheme, but for *this* theme push `red`'s hue to 10
  and pin `bg` to a literal hex." Currently the only options are editing the
  shared scheme (affects every theme using it) or copying the scheme wholesale.
- The forthcoming shell exposes `e` (edit `theme.toml`) as a first-class verb and
  needs theme-level derivation overrides to be real, distinct from the
  render-time hex pins in `overrides.toml`.

## The layering it establishes

```
schemes/<s>/scheme.toml    [mood] [hues] [fixed] [[list]]        base derivation
themes/<t>/theme.toml       scheme="s"  + [mood] [hues] [fixed] [[list]]   override
                                    │
                   merge_scheme_config(base, override)   ← per-key deep merge
                                    │
                    build_palette(raw, merged)  →  palette.toml   (COMPILE-time)
                                    ⊕ overrides.toml                (RENDER-time; untouched)
                                    │
                            effective palette → app fragments
```

**Two override layers in separate lanes — this is the load-bearing invariant:**

| Layer | File | Affects | Regenerated by |
|-------|------|---------|----------------|
| Derivation override | `theme.toml` (this change) | `palette.toml` derivation | recompile (re-extract + re-derive) |
| Concrete color pin | `overrides.toml` (exists) | final rendered colors | render (re-materialize) |

`theme.toml` overrides are **compile-relevant only**, exactly like `scheme.toml`:
they change *how the palette is derived from the wallpaper*, so they take effect
on recompile, not on a bare render. `overrides.toml` is never read or written by
this change.

## Merge semantics

`merge_scheme_config(base: dict, override: dict) -> dict` — a per-key deep merge,
**not** whole-section replacement:

- `[mood]` — merged key-wise. A theme setting `bg_lightness` keeps the scheme's
  `saturation_strength`, `hue_saturation_floor`, etc.
- `[hues]` — merged key-wise (per role). A theme setting `red = 10` keeps the
  scheme's other five hue anchors. Consistent with how `build_palette` already
  spreads `{**DEFAULT_HUES, **hues}`.
- `[fixed]` — merged key-wise (per role). A theme's fixed pin for a role overrides
  the scheme's pin for that role; other fixed roles are inherited.
- `[[list]]` — named arrays merged **by `name`**: a theme list with the same
  `name` as a scheme list replaces that list; a theme list with a new `name` is
  appended. (Arrays are positional in TOML but carry an identifying `name` field,
  so identity is by `name`, not index.)
- Any section absent from the override is taken from the base unchanged. Any
  section present only in the override is added.

Rationale for per-key over whole-section-replace: least surprising. A user who
nudges one hue does not expect the other five to vanish.

## Affected code

- `src/adorn/catalog.py`
  - Add `theme_overrides(theme_paths) -> dict`: parse `theme.toml`, return it with
    the `scheme` key removed (`{}` if the file is absent or has only `scheme`).
  - `theme_scheme` is unchanged.
- `src/adorn/compile.py`
  - Add `merge_scheme_config(base: dict, override: dict) -> dict` (above).
  - `compile_theme`: after `load_scheme_config(...)`, compute
    `merged = merge_scheme_config(scheme_cfg, catalog.theme_overrides(tp))` and use
    `merged` for both `build_palette(raw, merged, ...)` **and** the mood/strength/
    floor stat computation (so `CompileResult` reflects the effective config).
  - `build_palette` is unchanged — it already consumes a plain config dict.
- No change to `overrides.toml`, `render.py`, `commands.effective_palette`, or the
  `current` symlink mechanism.

## Backward compatibility

- A `theme.toml` containing only `scheme = "..."` yields an empty override dict, so
  `merge_scheme_config(scheme_cfg, {})` is `scheme_cfg`, and compilation is
  byte-identical to today.
- The `default` scheme and all existing themes are unaffected.
- No migration: the new sections in `theme.toml` are all optional.

## Testing (TDD, RED first)

New tests in `test_compile.py`:

- `merge_scheme_config` unit tests:
  - per-key `[mood]` merge (override one key, siblings inherited);
  - per-key `[hues]` merge (override `red`, other hues inherited);
  - per-key `[fixed]` merge;
  - `[[list]]` merge by name (same name replaces; new name appends);
  - empty override returns base unchanged (identity).
- `catalog.theme_overrides`:
  - returns `{}` for a `theme.toml` with only `scheme`;
  - returns the override sections (without `scheme`) when present;
  - returns `{}` when `theme.toml` is absent.
- `compile_theme` end-to-end: a theme whose `theme.toml` sets `[fixed] accent="#abcdef"`
  on top of a scheme that derives `accent` produces `palette.toml` with the pinned
  accent — proving the override reaches derivation.

Existing suite must stay green unchanged (back-compat property).

## Conventions

Python ≥ 3.11, src layout, `encoding="utf-8"`, TDD with `.venv/bin/pytest`. Commit
`feat:`/`refactor:`, no co-author trailer (matches repo history).

## Out of scope

- Any shell behavior (separate spec).
- Changes to `overrides.toml` / render-time pins.
- A CLI surface for editing `theme.toml` overrides (the shell's `e` will provide
  the ergonomic path; raw file editing already works).
